gin 框架学习文档

发表于2024-09-13|更新于2024-09-13
|阅读量:

Gin 是一个轻量级的 Go 语言 Web 框架，以其高性能和简单易用而闻名。下面我会简要介绍 Gin 框架的核心源码部分，尤其是引擎（Engine）和路由组（RouterGroup）的实现。

1. Engine结构体

Gin 最为核心的组件是Engine，它负责管理HTTP服务器的配置、路由树、路由组等信息。所有的路由处理、请求分发等都通过Engine来完成。

它的结构如下：

Engine结构体中，RouterGroup和trees是与路由管理相关的关键部分。

  1. RouterGroup：

    RouterGroup用于组织路由的结构体。它允许用户将路由按照组进行组织，从而方便管理（例如将API的不同版本或不同模块的路由分组）。RouterGroup可以嵌套，允许创建子组，子组可以继承父组的前缀和中间件。

    RouterGroup包含了前缀路径（basePath）和中间件（Handlers）。通过RouterGroup，可以为一组路由设置统一的前缀和中间件。

    1. Handlers:
      • HandlersChain 类型，表示中间件处理链。每个路由组都可以有一组中间件，这些中间件会在处理请求时依次执行。
      • HandlersChain 是一个 []HandlerFunc，即处理函数的切片。
    2. basePath:
      • 路由组的基础路径。所有在这个组中注册的路由都会以这个基础路径为前缀。
    3. engine:
      • 指向 Engine 结构体的指针。通过这个字段，RouterGroup 可以访问并操作 Gin 框架的核心引擎。
      • 它的存在使得 RouterGroup 可以调用 Engine 方法和属性。
    4. root:
      • 一个布尔值，指示这个路由组是否是根路由组用。

  2. trees：

    当在RouterGroup中注册一个新的路由时，该路由会被添加到对应的路由树中。

    trees是一个存储路由树的集合。Gin框架使用了基于前缀树（前缀树，也称为trie树）的路由机制。每个HTTP方法（GET、POST等）都有一个对应的路由树，用于存储和匹配相应方法的路由。

    trees字段是一个map，键是HTTP方法，值是对应的前缀树根节点。这个结构使得Gin在处理请求时，可以快速地根据请求方法找到相应的路由树，并在树中进行路由匹配。

  • 计算绝对路径：calculateAbsolutePath 方法计算完整的路径，内部做的事就是把路由组的基础路径和相对路径拼接起来

Engine.addRoute

这是注册路由的关键步骤。

本方法负责将指定的路径和处理函数链（handlers）添加到路由树中，这样就算注册完成了。

以下是对 addRoute 方法的详细解析：

  • 合法性检测：assert1 用于检查路径是否以 / 开头，HTTP 方法是否为空，以及处理函数链（中间件和最后真正handler都会在这里，没有中间件至少也要有一个最后的handler）是否为空。这些都是添加路由的基本前提条件。
  • 获取或创建路由树根节点：root := engine.trees.get(method) 获取对应 HTTP 方法的路由树根节点。如果该方法的路由树不存在，那就创建一个新的节点作为根节点，然后将其添加到 engine.trees 中。（实际上，创建路由树根节点的逻辑对每种HTTP方法只会创建一次，因为路由树是按照HTTP方法的种类开枝散叶的）
  • 添加路由：root.addRoute(path, handlers) 将路径和处理函数链添加到路由树中。具体实现中，这会将路径解析成节点，并构建一棵前缀树（Trie），以高效地匹配请求路径。这一步是本方法的核心步骤，接下来继续分析它。
  • 更新参数和路径段统计：countParams 和 countSections 分别用于统计路径中的参数数量和路径段数量，并更新 engine.maxParams 和 engine.maxSections。

3. Gin 与中间件

下面尝试在 Gin 框架里追踪中间件的相关逻辑。

3.1 定义路由组时——【中间件注册】

Gin 框架中，中间件的注册非常简单。像在上一章【Gin 框架的工作流程】的示例代码给出的那样，无论是针对全局还是特定的路由组，只需调用gin.Use函数即可完成注册。

正如一开始介绍Engine结构体时提到的，这里的Handlers是HandlersChain类型，而HandlersChain是RouterGroup路由组（根路由组也是一个路由组，这也是【针对全局】和【针对特定路由组】的中间件注册流程一致的原因）里的一个处理函数切片。

中间件是由整个路由组共享的，所以注册时只需要把新的中间件加在Handlers的最末尾即可。

3.2 定义路由时——【 HandlersChain 拼接】

一条路由的终点是一个Handler，但是实际上这个Handler本身和在它之前执行的若干中间件Handler并没有实质性的区别，引起它们不同的只是它们的相对位置。在 Gin 框架下定义路由，实际上就是将定义路由这个节点之前的若干Handler与当前节点的Handler组合在一起，把相对路径变成绝对路径并记录进路由树，由此也可得知路由树的数据结构是压缩前缀树。

3.3 服务器响应时——【 HandlersChain 遍历】

3.3.1 HandlerChain的顺序执行

默认情况下，服务器处理请求时HandlerChain的遍历是按照注册顺序执行的。

3.3.2 Context.Next()

Context.Next方法实现如下：

handlers是一个切片，index指示了当前执行handler的索引号。因为handlers是handler注册时直接在尾部添加形成的，因此简单通过 for 循环就可以实现顺序遍历。

由于Context本身会暴露于调用链路中，因此用户可以在某个handler中通过手动调用 Context.Next 的方式来打断当前handler的执行流程，提前进入下一个handler的处理中。

这里有一个很巧妙的设计：index是在整个遍历过程中唯一的且只会单调增加，所以当用户主动调用的Next方法后，index的大小已经满足不再执行后续handler的条件了。所以当前被打断的handler成为了最后一个被执行的中间件，原本顺序中它后面的handler不会被重复执行。

本质上Context.Next是一个压栈调用的行为，这就保证了一开始先在handleHTTPRequest调用过一次Context.Next以后，无论HandlersChain内部如何各种修改中间件的执行顺序，最后handleHTTPRequest调用的Context.Next一定会返回然后继续处理被中断的逻辑，那么最后真正给出响应的处理函数一定会被执行。
